## FINDINGS

### F-001: CRITICAL (fixed) — aisos status leaked DD_API_KEY

Already fixed in 04050020Z patch. Root cause: bash variable
expansion ${DD_API_KEY:-NOT SET} outputs the key value when set.
Lesson: never use bash default-value expansion for credential
presence checks.

### F-002: HIGH — sos-worker has no authentication

Anyone who knows the URL (sos.nobulai.tools) can POST fake
heartbeats or SOS broadcasts. No API key, no shared secret,
no verification. Combined with F-003, this is trivially exploitable
from any browser on the internet.

### F-003: HIGH — sos-worker CORS allows all origins

Access-Control-Allow-Origin: "*" means any website can make
requests to the Worker. An attacker could embed a script on any
page that floods the SOS endpoint with fake events.

### F-004: HIGH — aideploy prints raw API error responses

Lines 405, 439, 462, 570 print result.get('errors') and
result.get('message') directly. If Cloudflare or Vercel echo
back request headers (including Authorization: Bearer tokens)
in error responses, the credentials would be printed to the
terminal. This is the same class of bug as F-001.

### F-005: MEDIUM — claude-session-export prints Keychain stderr

Line 133 prints result.stderr.strip() from the macOS security
command. If the Keychain command outputs credential-adjacent
information on stderr, it would be printed. The risk is lower
(stderr from security CLI is typically error messages, not
secrets) but the pattern is unsafe.

### F-006: MEDIUM — claude-session-export prints credential keys

Line 141 prints the JSON key names from the Keychain credential
object: "Keys found: claudeAiOauth, ..." This discloses the
credential structure to anyone watching the terminal. Not the
secrets themselves, but information that helps an attacker
understand what credentials are stored.

### F-007: MEDIUM — sos-worker DD_SITE hardcoded

const DD_SITE = "us5.datadoghq.com" is hardcoded in the Worker.
If the DD site changes, the Worker must be redeployed. Should use
an environment binding (env.DD_SITE) with a fallback.

### F-008: LOW — aisos payload contains hostname and username

Heartbeat and SOS payloads include hostname, username, OS version,
architecture. This is by design (provenance), but if the Worker
has no auth (F-002), anyone can read /events and /latest to see
this information. Combined with open CORS (F-003), an attacker
can read it from any browser.

### F-009: LOW — session UUIDs in provenance headers

Every tool's provenance header contains the claude.ai session URL.
This is by design (provenance tracing) and the repo is public.
Not a vulnerability, but worth noting that session IDs are
permanently published.

### F-010: INFO — aipublish is clean

No credentials, no network calls, no user input handling beyond
file paths. No findings.

### F-011: INFO — aicatalog credential handling is clean

Keys go from os.environ.get() to HTTP headers. Never printed,
logged, or stored. Good pattern.

## PROPOSALS

### P-001: Add shared secret authentication to sos-worker

The Worker should check for an API key in the request header
(e.g., X-AISOS-KEY) and reject unauthenticated requests. The
key should be stored as a Cloudflare secret (env.AISOS_SECRET)
and set in aisos.sh as AISOS_SECRET env variable.

Priority: HIGH — must be done before deployment.

### P-002: Restrict CORS on sos-worker

Replace Access-Control-Allow-Origin: "*" with a whitelist:
nobulai.tools, *.nobulai.tools, localhost. Or remove CORS
entirely since aisos.sh uses curl, not a browser.

Priority: HIGH — must be done before deployment.

### P-003: Sanitize API error responses in aideploy

Never print raw API error responses. Extract only the error
code and a safe message. Pattern:

  log_err(f"Deploy failed (status {code}). Check credentials.")

NOT:

  log_err(f"Deploy failed: {result.get('errors', [])}")

Priority: HIGH — same class of bug as the credential leak.

### P-004: Sanitize subprocess output in claude-session-export

Replace line 133 with a generic error message:

  log_err("Keychain read failed. Is Claude Code installed?")

Do not print stderr from any subprocess that handles credentials.

Priority: MEDIUM.

### P-005: Remove credential key name logging

Replace line 141 with:

  log_err("Expected credential format not found in Keychain.")

Do not enumerate credential structure to the terminal.

Priority: MEDIUM.

### P-006: Env binding for DD_SITE in sos-worker

Replace hardcoded DD_SITE with:

  const DD_SITE = env.DD_SITE || "us5.datadoghq.com";

Priority: LOW.

### P-007: Rate limiting on sos-worker

Add per-IP rate limiting using Cloudflare's built-in rate
limiting or a simple KV-based counter. Prevent flood attacks.

Priority: MEDIUM — depends on P-001 (auth reduces the attack
surface significantly).

### P-008: Credential handling checklist (new decision)

Every tool, every mission, before delivery:
1. Search for any print/log statement that references credential
   variables (key, token, secret, password, auth)
2. Verify no raw API error responses are printed
3. Verify no subprocess stderr containing credentials is printed
4. Verify no bash variable expansion outputs credential values
5. Run the audit grep commands from this AAR

Proposed as D-018.

## RISK ASSESSMENT

The highest risk right now is deploying sos-worker without
auth (F-002) and with open CORS (F-003). If deployed as-is,
anyone can:
- Send fake SOS broadcasts (false alarms)
- Flood the KV store (denial of service)
- Read all heartbeat data including hostname and username
- Relay false events to Datadog through the Worker

DO NOT deploy sos-worker until P-001 and P-002 are implemented.

The aideploy error response printing (F-004) is the next
priority. It is the same class of bug that caused the credential
leak incident. Fix before the Commander runs aideploy against
live APIs.
