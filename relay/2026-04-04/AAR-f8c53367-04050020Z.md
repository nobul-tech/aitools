## INCIDENT

Commander's DD_API_KEY was printed in plaintext on the terminal
after running `aisos status`. The Commander initially attributed
the exposure to aideploy (which was run immediately after), but
investigation confirmed the source was aisos.

## ROOT CAUSE

aisos line 298 used bash variable expansion:
```
${DD_API_KEY:+set}${DD_API_KEY:-NOT SET}
```

When DD_API_KEY is set:
- `${DD_API_KEY:+set}` correctly outputs "set"
- `${DD_API_KEY:-NOT SET}` outputs THE ACTUAL KEY VALUE
  (because the variable IS set, so the default is not used)

Result: `DD_API_KEY: setACTUAL_KEY_HERE`

## WHO INTRODUCED THE BUG

Agent Fantastic. This tool was built and shipped in the
immediately prior mission. The credential exposure pattern was
not caught during code review, functional testing, or hash
verification. The Commander caught it by running the tool.

## FIX

Replaced with:
```
$([ -n "$DD_API_KEY" ] && echo 'set' || echo 'NOT SET')
```

Verified: prints "set" when key exists, "NOT SET" when absent.
No value leaked in either case.

## ADDITIONAL FINDING

claude-session-export (line 313) printed the last 8 characters
of the OAuth token: `Token: ...{token[-8:]}`. Partial credential
exposure. Fixed to print `Token: set` only.

## LESSONS

1. Never use bash variable expansion for credential presence
   checks. The `:+` and `:-` operators have subtle interactions
   that can leak values.
2. Credential exposure must be part of the code review checklist
   for every tool, every mission.
3. The Commander's testing caught what the agent's review missed.
   Surfacing Duty works both directions.
4. Security fixes ship immediately. No ceremony delay.
